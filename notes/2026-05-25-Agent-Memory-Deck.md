## 一、Agent 记忆的本质

**Agent Memory = 跨时间的意图-状态管理系统（Cross-Time Intent-State Management System）**

```
Intent（意图）→ State（状态）→ Memory（记忆携带）
```

1. **Intent starts incomplete** — 用户目标通常以"方向"而非完整规格到达
2. **State forms through work** — 澄清、决策、工具结果和产物形成工作状态
3. **Memory carries it forward** — 需要持久化的是"让下一轮或下一次会话能够继续"的状态

> 用户目标到达时往往不是完整规格，真正的工作状态是通过交互产生的，记忆的任务就是把这种状态带到下一步。

---

## 二、两个视角：Product View vs Engineering View

同一个记忆问题，有两个视角：

| 维度 | Product View（产品视角） | Engineering View（工程视角） |
|------|------------------------|---------------------------|
| **关注点** | 用户在意的连续性 | 行为需要生命周期管理 |
| **问题** | 当功能变成一层时 | 一旦记忆改变行为，就需要生命周期控制 |
| **关心什么** | 连续性、恢复、减少重复 | 分类、检索边界、上下文、控制 |

> **桥接点：** 一旦记忆改变行为 → 就需要生命周期管理。记忆先有连续性，然后进入生命周期和分层设计。

## 三、Agency 改变记忆形态（Product Memory）

### 模块化演进

| Agent 类型 | 记忆目标 | 核心问题 |
|-----------|---------|---------|
| **Chatbot** | 记住用户偏好、减少重复，让对话更连续 | 用户连续性 |
| **Coding Agent** | 保持项目规则、复用决策、跨会话恢复未完成工作 | 项目连续性 |
| **高阶 Agent** | 带回合适的状态，而非简单累积历史 | 选择性连续性 |

### 核心洞察

> **Agency 越高，记忆越从"流畅回复"变成"基础设施"。**
- Chatbot：优化用户连续性
- Coding Agent：优化项目连续性
- 自主 Agent：选择性连续性塑造行动

## 四、不同产品，相同连续性问题

| 产品 | 连续性类型 | 要记住什么 |
|------|-----------|-----------|
| **Chatbot** | 用户连续性 | 用户的偏好、闲聊上下文 |
| **Coding Agent** | 项目连续性 | Repo 的规则、架构模式 |
| **共享模式** | 选择性连续性 | 系统选择什么值得持久化 |

> 产品形态不同，但底层问题一致：**"什么能帮助下一次交互继续？"**

## 五、三个核心操作：Write → Retrieve → Revise

```
        Authorship（谁写）
              ↓
         Retrieval（谁取）
              ↓
         Revision（谁改）
```

### 1️⃣ Authorship（写入）
记忆可以由不同来源写入：
- **用户控制**：直接声明
- **项目文件**：从代码库自动提取
- **Agent 自写**：自动记录

### 2️⃣ Retrieval（检索）
存储的记忆**只有在当前步骤需要时才返回**。

> Retrieval is a **selection problem**（检索是一个选择问题）。

### 3️⃣ Revision（修订）
存储的记忆必须在**新证据使旧信息失效时更新**。

> Revision changes stored state after new evidence.

## 六、五个术语，五个不同的工作

| 术语 | 工作描述 |
|------|---------|
| **Memory** | 命名系统在时间上的记忆能力 |
| **Retrieval** | 系统可以执行的检索动作 |
| **Embeddings** | 将内容转换为向量 |
| **Vector Store** | 存储和搜索向量 |
| **Search** | 定位并返回有用状态 |

> 能力和操作不是同一个东西。工作流和基础设施也不是。**清晰的语言 → 清晰的设计。**

## 七、时间轴 × 内容轴（Two Axes）

### 时间轴：当前上下文 vs 持久化记忆
- **Current Context（工作暂存器）**：属于时间轴上的"活动上下文"
- **Persistent Memory（持久化记忆）**：需要进一步按内容分类

### 内容轴（持久化记忆内）：
- **Episodic（情景记忆）**：特定事件/会话
- **Semantic（语义记忆）**：概念、事实、知识
- **Procedural（程序记忆）**：过程、工作流、技能

> 工作暂存器属于时间轴上的活动上下文，不是第四种内容类型。

## 八、三层架构：Context → Prompt → Harness

```
┌─────────────────────────────────────┐
│          CONTEXT（上下文）           │ ← 决定模型看到什么
├─────────────────────────────────────┤
│          PROMPT（提示词）            │ ← 如何提问
├─────────────────────────────────────┤
│          HARNESS（框架）             │ ← 循环、工具和控制如何运行
└─────────────────────────────────────┘
```

- **Context** 决定模型看到什么
- **Prompt** 塑造请求方式
- **Harness** 支持递归和反馈循环

> 记忆跨越不止一层，因为它既改变模型看到什么，也改变系统行为如何。

---

## 九、记忆成为基础设施

### 演变路径

```
便利性 → 控制层 → 基础设施
```

1. **Convenience（便利性）** — 减少重复，让下一次交互更流畅
2. **Control（控制）** — 一旦 Agent 操作工具，记忆状态开始塑造系统"能安全做什么"
3. **Infrastructure（基础设施）** — Agent 依赖记忆来实现连续性、协调和受控的未来行为

> **强者取决于记忆状态。** 当 Agent 从回答问题转向执行操作时，记忆从便利功能变成控制层。

## 十、产品案例

### ChatGPT — 双层记忆
- **Saved Memory（保存记忆）**：稳定的事实，用户可见，跨会话持久化
- **Chat History（聊天历史）**：近期上下文，可演化，选择性返回

> 产品通过可视化边界（保存的记忆可查看/删除/管理），将"可持久性"与"近期上下文"分开。

### Claude Code — 项目连续性
1. **构建和测试命令** — Agent 需要可重复的命令来继续工作
2. **仓库约定** — 项目特定规则、架构模式、本地工作流
3. **先前的经验教训** — 曾尝试过的方案、已做出的决策、已知约束

> 编码 Agent 的连续性不仅仅是记住对话，更是记住项目的物理现实。

## 十一、行业方向：记忆正成为一层

| 方向 | 代表 | 描述 |
|------|------|------|
| **Universal Memory Layer** | Graph Context | 跨 Agent 的可复用连续性服务 |
| **Stateful Agent Platform** | Long-term memory OS | 将记忆转化为行为的平台 |
| **Memory Primitives** | Memory layer 框架 | 基础设施角色（框架、图、记忆 OS 思维） |

> 这些项目在范围上不同，但都汇聚在"记忆作为可复用系统行为"上。

---

## 总结

1. **记忆不是单个功能，而是跨时间的系统能力** — 需要从产品和工程两个视角理解
2. **Agency 越高，记忆越重要** — 从便利性到控制层再到基础设施
3. **三层架构（Context/Prompt/Harness）** 决定了记忆系统如何运作
4. **清晰分层（术语→操作→基础设施）** 避免设计混乱
5. **产品不同但问题相同** — 选择性连续性（Selective Continuity）
