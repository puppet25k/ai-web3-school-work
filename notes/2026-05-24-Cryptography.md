## 核心概念一览

| 概念 | 一句话 | 类比 | 安全红线 |
|------|--------|------|----------|
| 🔑 **私钥 (Private Key)** | 账户控制权本身 | 银行卡+密码+短信验证码合体 | ⛔ 绝不截图/上传/粘贴/发给 AI |
| 🔓 **公钥 (Public Key)** | 可公开，验证签名但不能反推私钥 | 你的收款二维码 | ✅ 可公开 |
| 📍 **地址 (Address)** | 不是身份，只是一个可验证控制关系的标识 | 银行账号 | ✅ 可公开 |
| ✍️ **签名 (Signature)** | 不是登录弹窗，是对消息/交易的授权证明 | 手写签名+指纹+印章 | ✅ 仅对特定消息 |
| 🔗 **哈希 (Hash)** | 固定长度指纹，验证数据一致性，不可逆 | 书的指纹 | ✅ 不可逆，无安全风险 |

## 详细理解

### 🔑 私钥 — 你拥有什么

- 一个 256 位的随机数（可以理解为 0 到 2²⁵⁶-1 之间的一个数）
- 私钥 = 账户的**完全控制权**
  - 控制资产转移
  - 控制合约调用
  - 控制身份验证
- **安全红线：** 永远不要以任何形式分享私钥
  - 不要截图、上传到云盘、粘贴到聊天工具
  - 不要发给 AI/LLM
  - 不要保存在联网的设备明文存储中
- 行业实践：用硬件钱包（Ledger/Trezor）或 MPC 方案管理

### 🔓 公钥 — 你可以分享什么

- 从私钥通过椭圆曲线加密（ECDSA）**单向推导**得到
- **可以公开分享** — 无法从公钥反推私钥
- 用途：验证签名
  - 别人用你的公钥，可以验证一条消息是否确实由你签名
  - 这不需要你暴露私钥

### 📍 地址 — 你被谁知道

- 从公钥通过哈希函数进一步推导得到（通常取公钥哈希的最后 20 字节）
- 常见形式：`0x...` 开头的 42 字符十六进制字符串
- **地址不是身份** — 它只是一个**标识符**
  - 一个私钥可以推导出多个地址（不同链、不同格式）
  - 地址本身不证明控制关系，证明控制关系的是签名
- 正确的理解方式：地址 = **一个可验证控制关系的标识**

### ✍️ 签名 — 你授权了什么

- 互联网的「登录」= 证明你知道密码
- Web3 的「签名」= 证明你控制某个私钥 + **你同意这条具体消息**
- 签名过程：
  1. 对消息计算哈希
  2. 用私钥对哈希进行加密
  3. 得到签名（一对数字 r, s）
- 验证过程（任何人可做）：
  1. 对原消息计算哈希
  2. 用公钥 + 签名解出哈希
  3. 对比两个哈希是否一致
- **签名是绑定到具体消息的** — 一个签名不能用于另一条消息

### 🔗 哈希 — 你验证了什么

- 任意长度的数据 → 固定长度的「指纹」
- 常见算法：SHA-256（比特币）、Keccak-256（以太坊）
- 三大特性：
  1. **确定性：** 相同输入 → 相同输出
  2. **单向性：** 从哈希值不可逆推原始数据
  3. **碰撞抵抗：** 找两个不同输入产生相同哈希值，在计算上不可行
- 应用场景：Merkle Tree（区块内交易的批量验证）、区块头链（Block Header Chain）

## 从密码学到区块链的逻辑链路

```
私钥 ──ECC──→ 公钥 ──Hash──→ 地址
  │                              │
  └──签名──→ 交易 ──Merkle──→ 区块 ──Hash──→ 区块链
```

- 密码学提供了**身份**（私钥/公钥对）和**授权**（签名）的数学基础
- 区块链用这些原语构建了一个无需信任第三方的价值转移系统
- 理解了密码学，就理解了 Web3 的信任模型：**不信任人，信任数学**

## 产品直觉总结

| 是什么 | 产品意识 |
|--------|---------|
| 私钥 | 控制权的终极凭证，所有 UX 围绕如何安全管理 |
| 公钥 | 分享没有风险，用来给人/机器验证你的身份 |
| 地址 | 是标识符不是身份，设计 UI 时不要混用 |
| 签名 | 是授权行为不是登录操作，设计要考虑 Gas 和无 Gas 签名 |
| 哈希 | 一切链上数据的指纹，可验证不可伪造 |
