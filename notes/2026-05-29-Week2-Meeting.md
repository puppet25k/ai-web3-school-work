## 二、黑客松方向：Agent 支付方案

### 讨论背景（源于 2758star 同学的笔记）

**Pact 思路启发：** 用户不是一次性把钱包完全交给 Agent，而是通过授权协议规定 Agent 能做什么、花多少钱、与谁交互、何时停止。

### 推荐黑客松方向：Agent Auto Top-up with Spending Guardrails

**场景：** Agent 使用 OpenAI/Claude/Zapier/Notion 等 API 或 SaaS 服务时，遇到额度不足、token 不够、订阅到期等问题，在用户预先授权范围内自动完成小额支付。

#### 优势
| 维度 | 说明 |
|------|------|
| 需求明确 | 余额不足就充值 |
| 风险可控 | 金额小、对象固定、频次有限 |
| 产品逻辑清晰 | 授权→执行→风控→通知 |
| 适合展示 | 可做出完整 Demo |

#### 关键展示点
不是"Agent 会付款"，而是 **"Agent 只能在被允许的范围内付款"**
- 单笔不超过 X 或 Y 美金
- 每天最多充值一次
- 总预算不超过 Z
- 超过预算必须人工确认

## 三、x402 + CAW / Pact：Agent 支付架构

（来自分享会的幻灯片讨论）

### 核心问题

> **问题不在会不会付，而在该不该付。**

### x402 协议
- 服务端返回 **402 Payment Required**
- 声明资源价格、收款方和付款要求
- Agent 付费后拿到真正结果
- → **解决"怎么付"**

### CAW / Pact
- 支付不应当盲付
- 限制 token、金额、频率、接收方
- 让支付先经过策略
- → **解决"该不该付、最多能付多少"**

### 审计链
- 付款前后的 request → pact → payment → delivery
- 都需要留下可回查记录
- 错误可追责、争议可仲裁

## 四、Harness 实操演示

### 核心概念（learn-claude-code 教程）

> **Agency comes from the model. An agent product = model + harness.**

| 概念 | 说明 |
|------|------|
| **Model** | 驱动力，来自训练 |
| **Harness** | 交通工具，Agent 产品的外壳 |
| **So1 - Agent Loop** | 核心循环：输入问题 → AI 思考 → 执行 → 检查 → 完成 |
| **So2 - Tool Use** | 在 So1 基础上加 4 个工具（read_file 等） |

### 关键观察：错误修复能力
```
AI说"运行 python hello.py"
→ 执行 python hello.py → 报错（命令不对）
→ AI看到错误，自动改成 python3 hello.py → 成功
→ 输出 "Done!"
```
