# 智能合约（Smart Contract）— 对照学习笔记

> **学习日期：** 2026-05-30
> **来源：** [AI x Web3 School Handbook — 智能合约（Smart Contract）](https://aiweb3.school/zh/handbook/web3/smart-contract/)
> **学习方法：** 与已知编程概念对照学习
> **标签：** #SmartContract #Solidity #EVM #ABI #Event #Gas

---

## 核心认知

> **智能合约不是"自动执行的法律合同"，而是部署在链上的程序。**
> 它把规则、资产和状态放到公开可验证的执行环境里，也把错误、权限和升级风险暴露给所有人。

**关键区别：**
- Web3 产品的很多核心逻辑都在合约里（token / NFT / DEX / 借贷 / 治理 / 质押 / 空投 / AA）
- 前端可以换，后端可以重构，但**合约一旦部署，修改成本和风险通常远高于普通应用代码**
- **不是学习立刻写复杂协议，而是先建立判断力**

---

## 四个对照概念

### 1️⃣ Contract ≈ 会话 + Java Class

合约的 `contract` 与已知概念的对照：

| 维度 | 智能合约（Contract） | 会话（Session） | Java Class |
|------|--------------------|----------------|-----------|
| **状态持久化** | 状态写入链上，全局可见且不可篡改 | 状态在内存中，请求结束即消失 | 状态在 JVM 堆中，进程结束即消失 |
| **调用方式** | 外部调用（EOA 或其他合约），需签名 + gas | 函数调用，无成本 | 方法调用，无额外固定成本 |
| **权限模型** | `msg.sender` + `modifier`，代码即规则 | 认证由框架/中间件控制 | 访问修饰符（public/private） |
| **部署后修改** | 理论上不可变（可升级需 proxy 模式） | 随时可以修改代码 | 可修改代码后重新部署 |
| **可见性** | **所有状态公开**，不是私有数据库字段 | 会话数据仅服务端可见 | 封装性由语言控制 |

**核心差异：** 合约的"公开可执行"特性带来了可组合性，也带来了攻击面。你写的不是普通函数，而是会管理真实资产和公共状态的程序。

**三个需要时刻注意的维度：**
- 🔍 **链上状态公开** — 很多人会看到
- ⚡ **执行环境特殊** — 受 gas、区块顺序和外部状态影响
- 🔐 **权限至关重要** — 谁可以 mint/pause/upgrade/withdraw，不能靠默认信任

---

### 2️⃣ Gas ≈ 底层语言的内存/CPU 开销

| 维度 | Gas | 内存/CPU 开销 |
|------|-----|--------------|
| **成本性质** | 每次执行消耗，直接关联经济成本（付 ETH） | 资源消耗，但通常没有直接经济成本 |
| **计费对象** | 用户/调用方支付 | 服务端/系统承担 |
| **限制方式** | Gas 上限（gas limit），超限 revert | OOM / CPU 超时 |
| **存储成本** | Storage 操作**非常贵**（SSTORE 20000 gas） | 磁盘/内存相对便宜 |
| **优化目标** | 减少链上存储，多用 event 替代 storage | 减少延迟、提高吞吐 |

**Gas 的影响：**
- 每次调用都付 gas → 代码效率直接影响用户体验
- Storage 贵，Memory 便宜 → 合约编程有独特的优化策略
- 失败时已消耗的 gas 不退 → 用户承担试错成本
- 类比：写合约就像在「每条指令都要花钱」的机器上编程

---

### 3️⃣ ABI ≈ API

| 维度 | ABI（Application Binary Interface） | API |
|------|--------------------------------------|-----|
| **作用** | 描述合约有哪些函数、参数、返回值、事件 | 描述服务端有哪些接口、参数、响应 |
| **格式** | JSON（函数签名 + 编码规则） | REST / GraphQL / gRPC |
| **用途** | 前端 SDK、脚本、区块浏览器、**AI Agent** 理解如何调用 | 前端、第三方应用调用后端 |
| **安全** | **只告诉你"能调什么"，不保证"调用是否安全"** | 通常有认证+鉴权层保护 |

**ABI 不告诉你的三件事：**
1. 参数的单位是不是 token decimals 后的整数
2. 当前账户是否有足够余额或 allowance
3. 函数是否被暂停或受权限限制

> **ABI 是机器可读接口，不是安全说明书。**

**对 AI x Web3 的意义：** ABI 是 Agent 理解合约能力的重要上下文。但 Agent 不能只靠 ABI 做决策，还需要链上状态验证和安全检查。

---

### 4️⃣ Event ≈ Log（日志）

| 维度 | Event（Solidity） | Log（传统编程） |
|------|------------------|----------------|
| **存储位置** | 写入链上交易收据（logs bloom），不占 storage | 写入文件/数据库/日志系统 |
| **可索引性** | 可被外部系统索引（通过 topics） | 通常全文搜索 |
| **访问方式** | 合约**不可直接读取**自己的 event | 程序可读取自己的日志 |
| **主要用途** | 让前端/索引器/分析工具追踪链上发生了什么 | 调试/审计/监控 |
| **成本** | 比 storage 便宜很多（LOG 操作 375 gas + 375/byte） | 几乎免费 |

**Event 的典型用途：** Transfer、OrderCreated、ParamsUpdated、PermissionChanged

**产品架构中的 Event：**
```
合约 emit event → 索引器捕获 → 数据库构建查询层 → 前端展示
```
很多产品页面并不是每次都直接扫链上 storage，而是通过**事件索引**构建更适合查询的数据层。

## 其他重要概念

### Solidity
- EVM 生态最常见的合约语言
- 看起来像普通编程语言，但运行在完全不同的环境
- 重点理解链上特有概念：**storage / msg.sender / modifier / event / external call / revert / 权限控制**

### EVM（Ethereum Virtual Machine）
- 运行智能合约的虚拟机
- Solidity → 编译为 EVM 字节码 → 部署到链上 → 节点执行
- 理解 EVM 有助于解释：为什么要 gas、为什么 storage 贵、为什么外部调用有风险、多链兼容

### 合约升级

| 维度 | 不可升级 | 可升级（Proxy） |
|------|---------|----------------|
| **优点** | 规则固定，信任成本低 | Bug 可修复，产品可迭代 |
| **缺点** | Bug 修复困难 | 引入管理员权限、治理攻击、用户信任问题 |
| **代表** | 早期 DeFi 协议 | 多数现代协议（通过 proxy + 多签 + timelock） |

**四个升级风险问题：**
1. 升级权限在谁手里？EOA / 多签 / DAO / timelock？
2. 用户能否提前看到升级提案和新地址？
3. 升级能否改变资产转移、提款、暂停或权限逻辑？
4. 密钥泄漏最坏结果是什么？

> 如果一个协议说"合约已审计"，但没有说清楚升级权限——**安全说明是不完整的。**

### 完整调用链路

```
用户点击按钮
  → 钱包展示交易信息（用户确认签名）
    → EVM 执行合约逻辑
      → 成功：更新状态 + emit event
      → 失败：revert
        → 前端监听回执，更新页面
          → 索引器读取 event，更新历史/看板
```

## 在 AI x Web3 中的位置

```
AI（理解和辅助）
  → 帮用户理解 ABI、生成调用参数、解释交易结果
  → 不把最终判断交给模型
  
钱包（确认和授权）
  → 确认什么可以执行

合约（最终规则和约束）
  → 可验证执行，错误 revert

监控系统（记录和审计）
  → 记录结果
```

> **AI 做建议和编排，钱包做授权确认，合约做可验证执行，监控系统记录结果。**  
> 这样即使 AI 输出不稳定，链上规则仍然有明确边界。

## 最小实践

1. 在区块浏览器里找一个已验证源码的简单 ERC-20 或 NFT 合约
2. 查看源码、ABI、事件和最近交易
3. 找出哪些函数会改变状态（需要签名+gas），哪些只读（不需要）
4. 找出合约的权限函数（owner / admin / pause / mint / burn / upgrade）
5. 找出最重要的 event，对应哪类用户动作
6. 用一句话解释：**这个合约最重要的风险边界是什么**
