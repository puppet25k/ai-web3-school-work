# 开发栈（Dev Stack）— 对照学习笔记

> **学习日期：** 2026-05-31
> **来源：** [AI x Web3 School Handbook — 开发栈（Dev Stack）](https://aiweb3.school/zh/handbook/web3/dev-stack/)
> **标签：** #DevStack #Remix #Hardhat #Foundry #OpenZeppelin #Viem #Wagmi

---

## 核心认知

> **Web3 开发栈不是一组随机工具名，而是一条从写合约、测合约、部署合约、连接钱包、调用合约到监控结果的工程链路。**

**开发栈的作用不是收集更多工具，而是把开发流程变成可重复的系统。**

一个项目至少要做到：
- ✅ 本地能跑
- ✅ 测试能复现
- ✅ 部署有记录
- ✅ 前端调用明确
- ✅ 合约地址可追踪

---

## 六步最小开发链路

```
写合约 → 编译合约 → 写测试 → 前端调用 → 部署 → 验证/监控
```

### 每一步的作用

| 步骤 | 做什么 | 缺了会怎样 |
|------|--------|-----------|
| **1. 写合约** | 在本地或浏览器 IDE 写出合约逻辑 | — |
| **2. 编译** | 得到 bytecode 和 ABI | 前端拿错 ABI → 参数编码错误 |
| **3. 写测试** | 覆盖核心状态变化和权限边界 | 只覆盖 happy path → 权限/失败分支漏掉 |
| **4. 前端调用** | 用合约地址和 ABI 读取或发送交易 | 地址没版本记录 → 用户不知和哪份合约交互 |
| **5. 部署** | 到测试网或主网 | — |
| **6. 验证 & 监控** | 区块浏览器验证源码、交易和事件 | 部署后不可追溯 → 排查成本极高 |

## 核心工具链

### 🔧 Remix — 合约实验台

**定位：** 浏览器里的 Solidity IDE。低门槛，适合入门、教学、原型快速验证。

**最适用的三件事：**
1. 快速复制一段 Solidity 代码，看能不能编译
2. 在 JavaScript VM 或测试网部署合约，观察构造函数、函数调用和 event
3. 用 Deploy & Run 面板理解函数调用参数和 gas 消耗

**不适合：** 长期替代工程化 repo。多人协作时应该进 Git + Hardhat/Foundry。

> Remix = "合约实验台"，不适合长期替代工程化仓库。

---

### 🔧 Hardhat — 合约工程框架

**定位：** 适合 JavaScript/TypeScript 项目，把合约开发接进测试、脚本和前端工程。

**典型 repo 结构：**
- `contracts/` — Solidity 代码
- `test/` — TypeScript 或 Solidity 测试
- `hardhat.config.ts` — 网络、编译器、插件和变量配置
- `artifacts/` — 编译生成的 ABI、bytecode 和 metadata

**优势：** 与 TypeScript、ethers、CI 的结合自然，适合与前端/后端/CI 紧密协作的项目。

---

### 🔧 Foundry — Solidity-native 测试工具

**定位：** 更偏命令行和 Solidity-native 测试，适合高强度合约开发和快速反馈。

**特点：**
- 用 Solidity 写测试（不需要 JavaScript）
- 运行速度快
- 支持 fuzz testing 和 fork testing
- 适合训练"先写测试，再改合约"的习惯

**常见命令入口：**
| 命令 | 作用 |
|------|------|
| `forge init` | 创建新项目 |
| `forge build` | 编译 |
| `forge test` | 运行测试 |
| `forge coverage` | 测试覆盖率 |
| `forge script` | 部署脚本 |

---

### 🔧 OpenZeppelin — 合约安全库

**定位：** 提供常用合约标准和安全组件，但不能替代对业务逻辑的审查。

**包含模块：**
- ERC-20 / ERC-721 / ERC-1155
- AccessControl / Ownable
- Pausable / ReentrancyGuard

**常见"用了库就安全"的错觉：**
> 你可以用 OpenZeppelin 的 ERC-20 实现 token，但仍然要自己决定谁能 mint、是否能 pause、owner 是否可以转移、升级权限是否有 timelock。这些不是库自动帮你做出的产品决策。

---

### 🔧 Viem & Wagmi — 前端链交互

**定位：** 解决前端与链交互的问题——读合约、写合约、管理账户、处理网络和缓存。

| 工具 | 特点 | 适用场景 |
|------|------|---------|
| **Viem** | TypeScript 原生，类型安全，底层调用能力强 | 非 React 项目、需要精细控制 |
| **Wagmi** | React Hooks，钱包连接、账户状态、合约读写 | React 应用快速开发 |

**前端接链最容易出问题的状态不一致：**
1. 钱包网络 vs 前端配置
2. 合约地址 vs 预期
3. RPC 返回值 vs 本地缓存
4. 交易 pending 状态同步

**前端至少需要分开的四类状态：**

| 状态 | 说明 |
|------|------|
| 🔵 **连接状态** | 钱包是否已连接、当前网络 |
| 📖 **读取状态** | 合约读取结果是否正在加载或已过期 |
| ✍️ **写交易状态** | 等待签名 → 已广播 → 等待确认 → 成功/失败 |
| ⚠️ **错误状态** | tx revert、网络错误、用户拒绝签名 |

> 如果这些状态混在一个按钮里，用户会很难判断自己是在"连接钱包""签名授权"还是"等待交易上链"。

## 最小投票合约实践流程

```
1. 写合约
2. 迁到 Hardhat/Foundry repo
3. 写测试：创建投票、重复投票、投票结束、非管理员操作失败
4. 用本地链部署，记录地址和 ABI
5. 用 Viem/Wagmi 写前端
6. 部署测试网 → 区块浏览器验证源码 → 检查 event
```

> 这个流程看起来比直接写页面调合约慢，但它能让错误更早暴露。Web3 项目真正耗时间的地方，往往不是写第一版，而是上线后发现合约地址错了、ABI 不匹配、权限没测、交易状态没处理。

## 在 AI x Web3 中的位置

**AI 能做的事：**
- 解释 ABI
- 生成部署脚本
- 补测试
- 排查交易失败
- 总结合约权限

**AI 参与的边界：**
> AI 参与开发栈后，验证流程反而要更明确。链上开发不是普通代码生成，高风险命令需要人工确认。

AI 生成的代码必须受到 repo workflow、测试、权限和 secret 管理约束。

---

## 最小实践

1. 用 Remix 部署一个极简计数器合约（含 `count` 和 `increment`）
2. 用 Hardhat 或 Foundry 建本地工程，写测试覆盖初始值和累加
3. 记录合约地址、ABI、部署网络、部署账号和交易哈希
4. 用 Viem 或 Wagmi 从前端读取 `count` 并发起 `increment` 交易
5. 在区块浏览器确认 event 被正确发出
6. 写下该链路中哪些信息必须进版本控制，哪些必须放在 `.env`
